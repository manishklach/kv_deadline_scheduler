The Problem

What LRU cannot see

Long-context LLM inference creates a memory scheduling problem that classical operating systems were never designed to solve. A single Llama-3-8B request at 128 k tokens occupies roughly 32–64 GB of KV cache. Under concurrent load, something must be evicted. The default answer is LRU — and it is wrong.

LRU sees timestamps and access counts. It has no idea that a block it just evicted belongs to a decode step scheduled to fire in 400 µs, that reloading it from DRAM costs 5 ms, or that the request's SLA allows only 1 ms of tail latency. That single bad decision blows the deadline entirely.

"Generic memory management asks: is this page hot? KV Deadline Scheduler asks: which block belongs to decode-critical state, how close is it to missing its deadline, and what is the cost of evicting it?"

A 5 ms NVMe reload against a 1 ms decode deadline is unrecoverable — tier depth matters

The fix requires the memory manager to know what the serving engine already knows: which blocks have deadlines, which are currently serving a decode step, and how expensive it would be to regenerate evicted state. That knowledge transfer is what this project formalises.

The Design

MemoryIntent — the core abstraction

Every KV block in the system carries a MemoryIntent object — a structured description that turns an anonymous page into a first-class scheduling citizen. The key fields: 

# src/kv_memory_intent/schema.py
@dataclass(slots=True)
class MemoryIntent:
    phase:           Phase    # PREFILL | DECODE | VERIFY | DONE
    priority:        Priority # COLD | WARM | HOT | DECODE_CRITICAL
    deadline_us:     int      # µs until block is needed (0 = no deadline)
    slack_us:        int      # µs before deadline becomes critical
    recompute_cost_us: int    # cost to regenerate if evicted
    pin_requested:   bool     # DECODE_CRITICAL auto-pins — invariant

    def __post_init__(self):
        # Safety invariant: DECODE_CRITICAL blocks cannot be accidentally evicted
        if self.priority == Priority.DECODE_CRITICAL:
            object.__setattr__(self, "pin_requested", True)

Formal ABI

Memory Intent ABI — the POSIX of KV scheduling

The project defines a formal, versioned C ABI that any LLM runtime (vLLM, TensorRT-LLM, SGLang, TGI) can implement to expose KV intent to the OS. This is the moat — whoever owns the ABI owns the ecosystem. 

// abi/memory_intent.h — version 1
typedef struct memory_intent {
    /* Identity */
    uint64_t object_id, request_id;
    uint32_t block_id; uint16_t object_type, abi_version;

    /* Placement */
    uint8_t phase;    // MI_PHASE_PREFILL | DECODE | VERIFY | DONE
    uint8_t priority; // MI_PRI_COLD | WARM | HOT | DECODE_CRITICAL
    uint8_t allowed_tiers; // MI_TIER_HBM | DRAM | CXL | NVME | REMOTE

    /* Scheduling */
    uint32_t deadline_us, slack_us;
    uint32_t recompute_cost_us, spill_cost_us;
    uint32_t flags; // PIN_REQUESTED | RECOMPUTE_OK | IS_DRAFT ...

    /* Disaggregation */
    uint32_t home_node_id, replica_node_id;
    uint64_t remote_addr;

    uint8_t _reserved[16]; // future extension — offsets never change
} __attribute__((packed)) memory_intent_t;

Paired with a lock-free shared-memory ring buffer (abi/memory_intent_ring.c) for low-latency per-step intent propagation, and a JSONL wire format for trace files. ABI versioning policy: fields append at end, existing offsets immutable forever.

The ring buffer allows a serving engine to push intent updates at decode-step granularity (every ~1 ms) with zero system call overhead between the producer and the OS-side consumer.

Research Extensions

Beyond single-node KV

Disaggregated scheduling

Mooncake, LMCache, and DistKV disaggregate KV cache across machines. When a block may live on a remote node, the deadline question expands: can we fetch it before it's needed, or is recompute cheaper than waiting for the network?

disaggregated/remote_intent.py adds network_rtt_us and home_node_id to the intent schema. RemoteAwarePolicy triggers local recompute when deadline_us < rtt_us × 1.5. The 3-node topology simulator (prefill / decode / storage) mirrors the exact Mooncake architecture.

Speculative decoding

Draft tokens may be rejected, making their KV blocks waste. No existing scheduler handles this. SpeculativeIntentPolicy tracks acceptance probability per draft node: blocks with acceptance_prob < 0.3 immediately downgrade to COLD. On VERIFY rejection, the entire draft subtree is freed immediately, recovering wasted HBM before the next step.

vLLM integration — zero code changes

integrations/vllm/vllm_kv_intent_plugin.py shows how to attach intent emission to a running vLLM instance using only the existing callback surface — no upstream vLLM patches required. Output writes to the memory_intent ring buffer, which the kernel-side consumer reads via /proc/kv_intent/.

Get Started

Zero runtime dependencies

Pure Python 3.11+, stdlib only. One command to see the core result.

# Install
git clone https://github.com/manishklach/kv_deadline_scheduler
cd kv_deadline_scheduler && pip install -e .

# Run the full policy sweep — 12× HBM pressure, all 5 policies × 5 profiles
python examples/benchmark_sweep.py \
  --seed 42 --requests 64 --blocks-per-req 24 \
  --block-size-mb 1 --hbm-mb 128 --dram-mb 4096

# Compare policies on your own request trace
kvmi import-request-trace \
  --requests examples/sample_request_trace.jsonl \
  --model llama-3-8b --out trace.jsonl
kvmi compare --trace trace.jsonl --hbm-mb 4096

# Run the tests
python -m pytest  # 62 passed
